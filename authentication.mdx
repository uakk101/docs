CodeCrafting Hub uses API keys to authenticate every request to the REST API. You pass your key as a Bearer token in the `Authorization` header — there are no sessions, cookies, or OAuth flows required for server-to-server access. Each key carries a scope that controls exactly which operations it can perform, so you can issue tightly scoped credentials to each integration or service without over-provisioning access.

## Get an API Key

Generate your API key from the CodeCrafting Hub dashboard. Each key is shown **only once** at creation time, so have a safe place ready to store it before you begin.

<Steps>
  <Step title="Open API Keys settings">
    Sign in to the [CodeCrafting Hub dashboard](https://app.codecraftinghub.com), navigate to **Settings**, then select **API Keys**.
  </Step>
  <Step title="Create a new key">
    Click **Create New Key**. Give the key a descriptive name that identifies the integration or service that will use it (for example, `billing-service-prod`).
  </Step>
  <Step title="Choose a scope">
    Select the scope that matches the minimum access your integration needs:

    - **read** — view-only access to patients, encounters, and claims
    - **read-write** — create and update records in addition to read access
    - **admin** — full access, including webhooks and organisation settings

    See [Key Scopes](#key-scopes) below for a full breakdown.
  </Step>
  <Step title="Copy and store your key">
    Click **Generate**. Copy the key immediately and store it in a secrets manager or environment variable.

    <Warning>
      Your API key is displayed **only once**. If you close this screen without copying it, you must revoke the key and create a new one — there is no way to retrieve it again.
    </Warning>
  </Step>
</Steps>

## Authenticate Requests

Include your API key as a Bearer token in the `Authorization` header of every request. Replace `YOUR_API_KEY` with the key you copied from the dashboard.

<CodeGroup>

```bash curl
curl https://api.codecraftinghub.com/v1/patients/PATIENT_ID \
  -H "Authorization: Bearer YOUR_API_KEY"
```

```python Python
import requests

headers = {
    "Authorization": "Bearer YOUR_API_KEY",
    "Content-Type": "application/json"
}

response = requests.get(
    "https://api.codecraftinghub.com/v1/patients/PATIENT_ID",
    headers=headers
)
patient = response.json()
```

```js Node.js
const response = await fetch('https://api.codecraftinghub.com/v1/patients/PATIENT_ID', {
  headers: {
    'Authorization': 'Bearer YOUR_API_KEY',
    'Content-Type': 'application/json'
  }
});
const patient = await response.json();
```

</CodeGroup>

## Key Scopes

Every API key is issued with one of the three scopes below. Choose the narrowest scope that still meets your integration's requirements — this limits the blast radius if a key is ever compromised.

| Scope        | Permitted Methods     | Access Level                                           |
|--------------|-----------------------|--------------------------------------------------------|
| `read`       | GET                   | View patients, encounters, and claims                  |
| `read-write` | GET, POST, PUT        | Create and update records, plus all read operations    |
| `admin`      | GET, POST, PUT, DELETE | Full access including webhooks and org settings       |

You cannot promote a key to a higher scope after it is created. If you need broader access, revoke the existing key and create a new one with the required scope.

## Rotate and Revoke Keys

Rotate your API keys on a regular schedule and immediately revoke any key that may have been exposed.

To **revoke** a key, go to **Settings → API Keys** in the [dashboard](https://app.codecraftinghub.com), locate the key by name, and click **Revoke**. Revocation is immediate — any request using that key will return a `401 Unauthorized` response straight away.

To **rotate** a key:

1. Create a new key with the same scope as the one you are replacing.
2. Update your integration or environment variable to use the new key.
3. Verify that requests are succeeding with the new key.
4. Revoke the old key.

Rotating in this order ensures zero downtime — your service is always authenticated while the swap takes place.

## Authentication Errors

The API returns standard HTTP error codes when authentication fails. Check the `error` field in the response body for a machine-readable reason and the `message` field for a human-readable explanation.

### 401 Unauthorized

A `401` response means the request arrived without a valid API key — either the `Authorization` header was missing entirely, the token was malformed, or the key has been revoked.

```json
{
  "error": "unauthorized",
  "message": "Invalid or missing API key",
  "status": 401
}
```

Verify that you are including the `Authorization: Bearer YOUR_API_KEY` header and that the key has not been revoked in the dashboard.

### 403 Forbidden

A `403` response means the API key is valid but its scope does not permit the requested operation — for example, using a `read` key to create a new patient record.

```json
{
  "error": "forbidden",
  "message": "Your API key does not have permission to perform this action",
  "status": 403
}
```

Check the [Key Scopes](#key-scopes) table and confirm that the key's scope covers the HTTP method and endpoint you are calling. If it does not, revoke the key and issue a new one with the appropriate scope.

## Security Best Practices

Protecting your API keys is critical in a healthcare environment where PHI is at stake. Follow these practices to reduce the risk of credential exposure:

- **Never embed keys in client-side code.** Any key included in browser JavaScript, a mobile app binary, or a public repository can be extracted and abused. Use API keys only in server-side or backend code.
- **Use environment variables.** Store keys in environment variables or a dedicated secrets manager (such as AWS Secrets Manager, HashiCorp Vault, or Azure Key Vault) rather than in configuration files checked into source control.
- **Scope keys to the minimum required access.** Issue a `read` key for read-only integrations and a `read-write` key only where record creation or updates are necessary. Reserve `admin` keys for infrastructure-level tooling.
- **Rotate keys regularly.** Establish a rotation schedule — quarterly at minimum — and rotate immediately any time a key may have been exposed through a log, an error report, or a misconfigured service.
- **Audit key usage.** Review the API Keys page in the dashboard periodically to identify and revoke keys that are no longer in use.
- **Use one key per integration.** Issuing a unique key to each service or environment (development, staging, production) lets you revoke a single compromised key without disrupting all other integrations.
