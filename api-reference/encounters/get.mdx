---
title: "GET /encounters/{id} — Retrieve a Clinical Encounter"
sidebarTitle: "GET /encounters/{id}"
description: "Fetch a single clinical encounter by its platform ID, with optional sideloading of related clinical notes and associated insurance claims."
---

You can retrieve the full details of any encounter your API key has access to by passing its platform identifier in the path. Use the optional `include` query parameter to sideload related records—such as clinical notes and claims—in a single round trip rather than making separate requests. You need at minimum the `read` scope to call this endpoint.

## Request

<ParamField header="Authorization" type="string" required>
  Bearer token for API authentication. Format: `Bearer YOUR_API_KEY`.
</ParamField>

<ParamField path="id" type="string" required>
  The platform identifier of the encounter to retrieve. Prefix: `enc_`.
</ParamField>

<ParamField query="include" type="string">
  Comma-separated list of related resources to embed in the response. Accepted values: `notes`, `claims`.

  Example: `include=notes,claims`
</ParamField>

<CodeGroup>
```bash cURL
curl 'https://api.codecraftinghub.com/v1/encounters/enc_01HXA1B2C3D4E5F6G7H8I9J0K?include=notes,claims' \
  -H 'Authorization: Bearer YOUR_API_KEY'
```

```python Python
import httpx

encounter_id = "enc_01HXA1B2C3D4E5F6G7H8I9J0K"

response = httpx.get(
    f"https://api.codecraftinghub.com/v1/encounters/{encounter_id}",
    headers={"Authorization": "Bearer YOUR_API_KEY"},
    params={"include": "notes,claims"},
)
encounter = response.json()
print(encounter["status"])
```
</CodeGroup>

<Tip>
  Sideloading `claims` is especially useful when you need to check the billing status of an encounter without a separate call to [GET /claims/{id}](/api-reference/claims/status).
</Tip>

## Response

A successful request returns HTTP **200 OK** with the encounter object. When `include` parameters are provided, the response also contains the nested `notes` and/or `claims` arrays.

<ResponseField name="id" type="string">
  Platform-generated encounter identifier. Prefix: `enc_`.
</ResponseField>

<ResponseField name="patient_id" type="string">
  The patient this encounter belongs to.
</ResponseField>

<ResponseField name="provider_npi" type="string">
  NPI of the rendering provider.
</ResponseField>

<ResponseField name="service_date" type="string">
  Date of service in `YYYY-MM-DD` format.
</ResponseField>

<ResponseField name="encounter_type" type="string">
  The encounter category (e.g., `office_visit`, `telehealth`).
</ResponseField>

<ResponseField name="place_of_service" type="string">
  CMS POS code for the care setting.
</ResponseField>

<ResponseField name="status" type="string">
  Current lifecycle status: `draft`, `in_progress`, `completed`, or `billed`.
</ResponseField>

<ResponseField name="diagnoses" type="array">
  Ordered list of diagnosis objects attached to the encounter.
</ResponseField>

<ResponseField name="procedures" type="array">
  Ordered list of procedure objects attached to the encounter.
</ResponseField>

<ResponseField name="notes" type="array">
  Clinical notes linked to this encounter. Only present when `include=notes` is requested.
</ResponseField>

<ResponseField name="claims" type="array">
  Claims submitted against this encounter. Only present when `include=claims` is requested.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 timestamp of encounter creation.
</ResponseField>

<ResponseField name="updated_at" type="string">
  ISO 8601 timestamp of the most recent update.
</ResponseField>
